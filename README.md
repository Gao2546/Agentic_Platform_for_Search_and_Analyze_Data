### About the project

An application designed to define the scope for companies to track news, stock prices, financial statements, and financial health. It utilizes AI to predict a company's financial status or upcoming events.

    - 2. Data Orchestration & Compute Engine (The Core Pipeline)
        
        Apache Airflow (The Orchestrator / DAG Scheduler)
        
        Acts as the central controller for all Workflows, managing Schedules and orchestrating downstream Services in proper sequence.
        
        Apache Spark (The Compute Engine / Distributed Processing)
        
        Awaits commands from Airflow to pull raw data for rapid Distributed ETL/ELT operations.
        
        Apache Kafka or RabbitMQ (Message Broker & Event Streaming)
        
        - *Role:* While Airflow handles Batch tasks (e.g., pulling quarterly financial statements), the Streaming system takes over operations requiring ultra-low latency, such as monitoring Price Action for XAU/USD, BTC/USD, or Forex down to the minute or second.
        - *Workflow:* AI-generated Tool Scripts for price monitoring will Publish data into the Message Broker when specific conditions are met (e.g., a reversal candlestick forms or a support level breaks). The system will Trigger alert models or execute Module A instantly, bypassing the scheduled Airflow cycle.
        
        **The Triggering Mechanism (API Endpoint)**
        
        The bridge between Streaming Scopes and Batch Scopes operates through a central Backend API Endpoint (e.g., `/api/v1/schedules/trigger/{schedule_id}`). The Backend translates this Request into a REST API command to spawn a new Apache Airflow DAG Run, attaching precise configurations via the `arguments` variable defined in the PostgreSQL `tasks` table.
        
    - 3.A Workflow Batch Tasks (Controlled by Airflow)
        
        Task 3.1: Searching and Collect Data
        
        Airflow triggers a Tool Script (e.g., Python or Go code fetching stock/news APIs).
        
        Ingests raw data into MinIO.
        
        Task 3.2: ETL/ELT Task (Spark Processing)
        
        Airflow commands Spark to initialize.
        
        Spark fetches raw data from MinIO, cleans it, and transforms the format.
        
        Saves the cleaned data back into MinIO in **Apache Parquet** or **Delta Lake** formats (maximizing performance and preserving Data Schemas without needing Data Connector API conversions), then loads essential data into PostgreSQL/MongoDB.
        
        Task 3.3: Hybrid Analytics & Prediction Task
        
        In this phase, Apache Airflow checks the "Config" defined by the User or AI within the Schedule to determine the required processing Module. This is divided into 2 Modules:
        
        Module A: Traditional Logic & Calculation (Standard Code)
        
        Ideal for operations with fixed formulas or rigid conditions that do not require predictive modeling, delivering 100% accuracy and high-speed processing.
        
        - **Engine:** Python, Go, or C++ code (Existing Tool Scripts or those synthesized by AI).
        - **Use Cases:**
            - **Technical Analysis:** Calculating RSI, MACD, Moving Averages, or identifying reversal points from Candlestick Patterns using historical price data.
            - **Financial Ratios:** Calculating financial metrics from balance sheets (e.g., P/E, P/BV, D/E Ratio).
            - **Rule-based Alerts:** Scanning data for specific conditions, such as "Trigger an alert if the price drops by more than 5% within 1 hour."
        
        Module B: AI & Machine Learning Inference
        
        Ideal for analyzing unstructured data or discovering complex patterns beyond the capabilities of standard conditional logic.
        
        - **Engine:** LLMs (e.g., GPT-4, Llama 3), Time-Series Forecasting Models, or Classification Models.
        - **Use Cases:**
            - **Sentiment Analysis:** Tasking an LLM to read economic news or meeting minutes and classify the sentiment as "Bullish" or "Bearish."
            - **Predictive Modeling:** Forecasting the probability of a company announcing profit growth in the next quarter.
            - **Event Extraction:** Extracting critical data points from financial reports (e.g., dividend payout dates, executive resignations).
        
        Task 3.4: Data Visualizations/Summarize
        
        Fetches data from MongoDB/PostgreSQL to generate Summary Tables, preparing them for Frontend rendering.
        
    - 3.B Workflow Streaming Tasks (Event-Driven Pipeline) (Controlled by RabbitMQ or Apache Kafka)
        
        **Task 3.1: Continuous Data Ingestion (WebSocket & Stream)**
        
        - **Workflow:** Instead of time-based scheduling via Airflow, the system spawns a Streaming Worker Container running a long-running process (e.g., in Go, Rust, or Python) to maintain an active WebSocket or gRPC connection.
        - **Process:** The script receives real-time (Tick-by-Tick) price or news data instantly as the market moves, immediately shooting the raw data into the **Message Broker (e.g., RabbitMQ or Apache Kafka)**. Simultaneously, an asynchronous process writes the raw data to MinIO as a backup.
        
        **Task 3.2: Real-time Stream Processing (In-memory ETL)**
        
        - **Workflow:** A secondary set of Workers (potentially utilizing Apache Spark Structured Streaming or optimized Go/Rust Workers for maximum speed) Subscribes to the Message Broker Topic.
        - **Process:** Performs On-the-fly data cleaning (e.g., filtering out erroneous data) and rapid Aggregation, such as compiling 1-second ticks into 1-minute or 5-minute OHLCV candlestick data, then routing the cleaned data back into a new Message Broker Topic pending analysis.
        
        **Task 3.3: Event-Driven Analytics & Prediction Task**
        The Message Broker subsequently distributes the data to Workers for processing according to User Configs, still leveraging the 2 Modules but operating in Real-time:
        
        - **Module A: Traditional Logic (Fast Lane)**
            - *Ideal for:* Computationally intensive tasks requiring Low Latency.
            - *Engine:* Go, Rust, or C++ code (user-uploaded or selected from a library) intercepting incoming Stream data.
            - *Use Cases:* Upon receiving a new candlestick, the system instantly calculates RSI and MACD. If conditions are met (e.g., RSI drops below 30 or a reversal pattern emerges), the system immediately fires a Rule-based Alert or triggers an API to execute a trade.
        - **Module B: AI & Machine Learning Inference (Heavy Lane)**
            - *Ideal for:* Deep analysis triggered by anomalous events detected in Module A.
            - *Engine:* LLMs or Time-Series Models pre-deployed as internal APIs.
            - *Use Cases:* If Module A detects a severe, anomalous drop in the BTC/USD price (Event Trigger), the system instantaneously commands Module B to pull the latest news headlines from the News Stream and perform Sentiment Analysis via LLM to ascertain the negative news driving the drop, validating the trade signal.
        
        **Task 3.4: Live Broadcasting & State Update**
        
        - **Process:** Instead of building a Summary Table and waiting for the Frontend to Pull the data, the system utilizes the Message Broker to route analytical results to a WebSocket Server, Pushing the data directly to the user's Dashboard. This enables live updates for charts and alerts.
        - **Storage:** The Latest State is "Upserted" (updated or inserted) into **MongoDB**, ensuring the system always has the most current data ready if a user opens a new Web Application session.
        
    - 3.C Workflow Mix Tasks (Hybrid Event-Triggered Pipeline)
        
        **Concept:**
        A hybrid approach that shatters the limitations of traditional pipelines by allowing individual Scopes to operate independently. Tasks can be assembled modularly (it is not mandatory to have all 4 Tasks; if the terminal Visualize Task is omitted, the system automatically hides that UI component).
        
        This architecture leverages a **Cross-Pipeline Triggering mechanism via API**, seamlessly marrying the ultra-low latency of Streaming pipelines with the profound analytical depth of Batch LLM processing.
        
        **Cross-Scope Execution Example:**
        • **Scope 1: The Background Data Collector (Batch Schedule)**
            ◦ **Role:** Fetches news data at predefined intervals (e.g., every 1 hour).
            ◦ **Workflow:** Airflow triggers Task 3.1 to pull news from an API $\rightarrow$ forwards to Task 3.2 for ETL text cleaning $\rightarrow$ saves to MinIO and generates Vector Embeddings into PostgreSQL `knowledge_embeddings`.
            ◦ **State:** Runs independently in continuous cycles.
        
        • **Scope 2: The Watchdog (Streaming Schedule)**
            ◦ **Role:** Real-time asset or stock price surveillance.
            ◦ **Workflow:** Task 3.1 maintains an open WebSocket for continuous price intake $\rightarrow$ Task 3.3 (Module A) applies Traditional Logic to detect anomalies (e.g., volume spikes or critical support breakdowns).
            ◦ **The Trigger:** Upon detecting an anomaly, the system bypasses the wait for the next Batch cycle and fires an **HTTP POST Request (API)** containing a Context Payload (e.g., `{"ticker": "AAPL", "timestamp": "...", "event_type": "BREAKOUT"}`) to instantly Trigger Scope 3.
        
        • **Scope 3: The Deep Analyzer (Triggered Batch DAG)**
            ◦ **Role:** Executes strictly Once per trigger (Run Once), acting as the central "brain" for final decision-making.
            ◦ **Workflow:**
                ▪ Receives the Context Payload from Scope 2.
                ▪ Skips Task 3.1 and 3.2, jumping directly into **Task 3.3 (Module B: AI Inference)**.
                ▪ The LLM leverages Tools to query the pre-analyzed price data from MongoDB and utilizes RAG to pull the latest news context from Scope 1. It synthesizes this data to evaluate the root cause of the price drop and dictates the appropriate operational response.
                ▪ **Task 3.4 Visualize:** Dispatches a comprehensive summary report—integrating charts and text—as a Dashboard notification or mobile application alert.
        
        **The Triggering Mechanism (API Endpoint)**
        The bridge between Streaming Scopes and Batch Scopes operates through a central Backend API Endpoint (e.g., `/api/v1/schedules/trigger/{schedule_id}`). The Backend translates this Request into a REST API command to spawn a new Apache Airflow DAG Run, attaching precise configurations via the `arguments` variable defined in the PostgreSQL `tasks` table.
        
    - 4. CI/CD & AI Sandboxing (Security and Automation)
        
        **GitHub Actions:** Code pushes (PySpark, Airflow DAGs) automatically trigger a Docker Image Build and Push to the Registry.
        
        - **Pipeline to validate AI Tool Scripts**
            - 1. Pre-validation Pipeline (Static Code Analysis & Security Check)
                
                **Static Code Analysis & Security Check (Pre-execution)**
                
                - Before the system provisions an Isolated Docker Container to Build and execute an AI-generated Tool Script (AI Schedule), an initial static code validation layer is introduced to conserve resources and prevent crashes:
                    1. **Syntax & AST Check:** Utilizes modules like `ast` (for Python) to verify structural integrity and successful compilation.
                    2. **Linter & Malicious Pattern Scanner:** Scans for unauthorized Library Imports (e.g., `os.system`, `subprocess` which could compromise the system) or detects unconditional Infinite Loops.
                    3. **Language-Specific Compiler Check:** If the AI synthesizes low-level code (e.g., Go or Rust), a dry-run utilizing `go build` or `cargo check` is executed in a restricted environment. Any Compiler Errors are fed back to the AI for immediate Self-correction, avoiding the overhead of launching a live container.
            - **2. Pipeline AI Code Sandbox**
                
                Newly generated AI Tool Scripts (AI Schedule) are initially executed inside an Isolated Docker Container to ensure functional viability and system safety. Upon successful Validation, the path is committed to PostgreSQL, making it available for Airflow orchestration.
                
                **Resource & Time Limits:** Executing AI-synthesized code within Docker Containers carries inherent risks of Infinite Loops or Memory Leaks. *Recommendation:* Strictly enforce Docker `cgroups` (e.g., capping RAM at 512MB, CPU at 1 Core) and implement strict Timeouts (e.g., immediately killing the Process if the Script exceeds 60 seconds).
                
                **Network Restriction:** Sandbox code must be denied direct internal Database access (Postgres/Mongo). Network access should be exclusively restricted to outbound Internet traffic for external API consumption.
                
            1. If validation fails, Error logs must be routed back to the LLM model to refine and improve the code.
